# Keycloak integration test fixture

A local, reproducible [Keycloak](https://www.keycloak.org/) provider for
`py-identity-model` integration tests. Mirrors the structure of the
`node-oidc-provider` fixture: `docker compose up` builds an image that bakes the
committed realm export in and starts Keycloak with `--import-realm` — no manual
console setup, everything is committed.

The realm is `COPY`d into the image via the `Dockerfile` rather than
bind-mounted at runtime, because a single-file bind mount is unreliable across
Docker backends (rootless / VM-backed daemons can turn the file path into an
empty directory, so `--import-realm` silently imports nothing).

## Start / stop

```bash
# from this directory
docker compose up -d --build --wait   # build image, boot, block until realm is ready
docker compose down                   # stop and remove the container
```

The `--wait` flag honours the healthcheck, which probes the imported realm's
discovery document (not just the open port), so it only returns once
`start-dev --import-realm` has finished loading `realm-export.json`.

## Endpoints

Realm: **`py-identity-model`** — issuer `http://localhost:8080/realms/py-identity-model`.

| Purpose | URL |
| --- | --- |
| Discovery | `http://localhost:8080/realms/py-identity-model/.well-known/openid-configuration` |
| JWKS | `http://localhost:8080/realms/py-identity-model/protocol/openid-connect/certs` |
| Token | `http://localhost:8080/realms/py-identity-model/protocol/openid-connect/token` |
| Authorize | `http://localhost:8080/realms/py-identity-model/protocol/openid-connect/auth` |
| UserInfo | `http://localhost:8080/realms/py-identity-model/protocol/openid-connect/userinfo` |
| End session | `http://localhost:8080/realms/py-identity-model/protocol/openid-connect/logout` |
| Admin console | `http://localhost:8080/` (admin / admin) |

The realm's RSA signing keys are **generated by Keycloak on import** (not
committed), so tests must fetch JWKS live from the `certs` endpoint rather than
pinning a key.

## Clients

| clientId | Type | Secret | Flows | Notes |
| --- | --- | --- | --- | --- |
| `py-identity-model-client` | confidential | `py-identity-model-client-secret` | client_credentials (service account), device grant, token exchange | primary `TEST_CLIENT_ID`; `oauth2.device.authorization.grant.enabled` + `standard.token.exchange.enabled` |
| `test-auth-code` | confidential | `test-auth-code-secret` | authorization_code, direct grant | back-channel logout URL + `session.required=true` configured for logout tests |
| `test-pkce-public` | public | — | authorization_code (PKCE **S256 required**) | redirect `http://localhost:8080/callback` |

Redirect URI for the interactive clients is `http://localhost:8080/callback`
(matches the node-oidc fixture). Keycloak intercepts the callback with its own
login page — the auth-code test login helper is wired in KC.3.

## Test user

`test-user` / `test` — email `test@example.com` (verified), name "Test User".
Needed for the auth-code login form (Keycloak has no auto-login equivalent to
node-oidc `devInteractions`). The password matches the shared
`AuthCodeFlowConfig` default so the login helper needs no per-provider config.

## Capabilities

- **JWT access tokens** — Keycloak issues signed JWT access tokens by default,
  so the JWT-validation tests exercise real tokens.
- **RP-initiated logout** — `end_session_endpoint` is advertised in discovery.
- **Back-channel logout** — configured on `test-auth-code`
  (`backchannel.logout.url`, `backchannel.logout.session.required=true`) for the
  BCL / RPL / KC.5 live tests.
- **Standard scopes** — `openid`, `profile`, `email` (plus Keycloak's built-in
  `roles`, `web-origins`, etc.) are auto-created on import; the realm export
  intentionally does not declare `clientScopes` so the built-in defaults are
  preserved (see DEC-003).
- **Opaque tokens** — Keycloak emits JWTs, not opaque tokens, so the
  introspection / opaque-token integration tests **skip cleanly** (no
  `TEST_OPAQUE_CLIENT_ID` is configured for this provider).
- **Device authorization grant** (RFC 8628) — enabled on
  `py-identity-model-client` via `oauth2.device.authorization.grant.enabled`.
- **Token exchange** (RFC 8693) — standard token exchange enabled on
  `py-identity-model-client` via `standard.token.exchange.enabled` (a
  supported feature in Keycloak 26.2+, no server feature flag required).
- **private_key_jwt** — advertised in discovery, but the live
  `private_key_jwt` tests pin the static key registered only in the node-oidc
  fixture, so they **skip cleanly** against Keycloak.

## Wiring (added in later stories)

This directory (KC.1) is the fixture only. The `.env.keycloak` config and the
`make test-integration-keycloak` target are added in **KC.2**; `provider_matrix.py`
picks the provider up automatically from `.env.*`. The full capability-gated
suite runs in **KC.3**, and the CI job in **KC.6**.
